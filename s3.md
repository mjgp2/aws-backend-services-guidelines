## 1. Purpose

These are the S3 defaults I recommend for backend services.

S3 is the default place for large binary payloads, exports, imports, media,
documents, and any immutable object that does not belong in PostgreSQL.

## 2. Core Rules

- store large object bytes in S3, not PostgreSQL
- keep metadata, ownership, and workflow state in PostgreSQL
- design object keys for operability and lifecycle management, not human
aesthetics
- favor immutable object writes over in-place mutation
- make storage-class and lifecycle choices explicit
- treat egress cost and accidental retention as first-class risks

## 3. Bucket Design

### 3.1 Default posture

Start simple:

- prefer a small number of purpose-specific buckets rather than one bucket per
feature
- separate materially different security or retention domains into different
buckets

Good reasons to split buckets:

- different data-classification levels
- different retention or deletion rules
- different public-delivery posture
- different replication or legal-hold requirements

Buckets should also carry the standard application tags from
[Infra]({{ '/infra/' | relative_url }}) so storage
cost and application ownership stay attributable through the shared tagging and
metadata model.

### 3.2 Key layout

Object keys should be:

- deterministic where deduplication or idempotency matters
- prefix-structured by data family or lifecycle domain
- safe to archive, backfill, and delete in batches

Example shapes:

- `uploads/{tenant_id}/{yyyy}/{mm}/{uuid}`
- `objects/sha256/{sha256}`
- `exports/{tenant_id}/{job_id}/{filename}`
- `events/{yyyy}/{mm}/{dd}/{uuid}.json`

Do not make filenames the primary identity when the system needs strong
idempotency or deduplication.

## 4. Default Storage Class Policy

### 4.1 Default write class

Default to `INTELLIGENT_TIERING` unless there is a clear reason not to.

Arguments for `INTELLIGENT_TIERING` as the default:

- good default for long-lived objects with uncertain access patterns
- moves storage optimization closer to the platform instead of relying on
  manual class selection later
- reduces the chance that teams leave aging data in `STANDARD` forever just
  because no one revisits the lifecycle policy

### 4.2 When to use `INTELLIGENT_TIERING`

Use `INTELLIGENT_TIERING` when:

- the object is expected to live longer than a short temporary workflow
- objects are long-lived
- access patterns are unpredictable
- the object count and monitoring overhead are acceptable

Avoid it when:

- objects are short-lived
- objects are temporary enough that they will be deleted before tiering value
  matters
- retrieval patterns are already well understood
- monitoring and per-object charges erase the benefit
- the bucket holds many small objects — objects under `128 KB` are always billed
  at S3 Standard rates regardless of storage class, so the tiering benefit does
  not apply to them

Cost reality check:

- `INTELLIGENT_TIERING` adds a monitoring and automation charge per object; for
  buckets with millions of small objects this fee can exceed the storage
  savings entirely
- objects below `128 KB` are charged at Standard rates even when stored in
  `INTELLIGENT_TIERING` — it is the wrong class for small-object-heavy buckets
- unlike `STANDARD_IA` and `ONEZONE_IA`, there is no minimum storage duration
  in `INTELLIGENT_TIERING`; objects can be deleted at any time without
  early-deletion charges

Operational reminder:

- if `INTELLIGENT_TIERING` is the default, writers need to set the storage
  class at upload time unless the bucket policy, SDK wrapper, or platform layer
  already does it for them
- do not assume objects will land in `INTELLIGENT_TIERING` automatically if the
  write path still defaults to `STANDARD`

### 4.3 When to use infrequent-access classes

Consider `STANDARD_IA` or `ONEZONE_IA` only when:

- objects are clearly cold
- retrieval latency and retrieval fees are acceptable
- the business is intentionally trading convenience for lower storage cost

Default stance:

- prefer `STANDARD_IA` over `ONEZONE_IA` unless the lack of cross-AZ
  redundancy is explicitly acceptable: `ONEZONE_IA` stores data in a single
  AZ, so a catastrophic failure of that AZ means permanent data loss —
  this is a resilience tradeoff, not merely a durability rating difference
- both `STANDARD_IA` and `ONEZONE_IA` have a `30`-day minimum storage
  duration; objects deleted before `30` days are billed for the full period
- do not move customer-critical canonical data into these classes just to save
  storage cost

### 4.4 Glacier classes

Use Glacier-style classes for compliance archives, long-tail forensic
retention, and backups or exports that are rarely read. Do not place
user-facing active product data in Glacier retrieval flows unless the product
explicitly promises archive-style access.

## 5. Lifecycle And Retention

Every bucket or prefix family should have an explicit lifecycle policy.

Each object family should define:

- whether it is canonical, derived, temporary, or archived
- the retention period
- the deletion trigger
- whether deletion is immediate, delayed, or blocked by legal/compliance rules
- which job, lifecycle rule, or operator-owned process enforces cleanup

Typical lifecycle rules:

- expire incomplete multipart uploads
- delete abandoned temporary uploads
- transition cold data to a cheaper storage class only when the economics are
clear
- delete short-lived exports and generated files automatically
- keep canonical product data until a defined product or compliance rule says
otherwise

Default rule:

- temporary objects should have aggressive expiry
- generated artifacts should have explicit retention
- canonical source objects should have deliberate, reviewable retention policy

Deletion posture:

- temporary uploads should be deleted automatically
- derived or regenerable objects should not outlive their product value by
default
- object deletion should be coordinated with database metadata cleanup so the
system does not accumulate broken references or orphaned storage
- legal-hold or compliance-retained objects should be isolated clearly from
normal product lifecycle rules

Object tagging is optional by default, but useful when:

- lifecycle rules need to distinguish object classes within one bucket
- storage cost needs to be attributed more precisely than bucket-level tags
- compliance or legal-hold flows depend on object-level markers

Do not rely on object tagging for core identity if the same distinction can be
expressed more simply through bucket or key-prefix design.

## 6. Access Patterns

### 6.1 Service access

- backend services should access S3 with IAM roles, not static credentials
- writes should be scoped to the minimum required prefixes and buckets
- cross-service access should be explicit and narrow

### 6.2 Client access

- prefer presigned URLs or CloudFront signed access for direct client reads and
  writes
- keep signed URL TTLs short
- keep public buckets as the exception, not the default
- separate control-plane auth from object transfer where that improves scale
- authorize the caller against control-plane metadata before issuing a signed
  URL; knowledge of an object hash or object key is not permission by itself

CloudFront posture:

- direct S3 access is the default when the object path is private, mostly
  uncacheable, and does not need an extra edge control point
- use CloudFront when objects are materially cacheable, or when you
  specifically need edge controls such as WAF at the edge, custom-domain
  delivery, or hiding the S3 origin behind one public entrypoint
- do not add CloudFront by default for mostly uncacheable objects unless you
  specifically need those edge features

Before launch, the client-facing object path needs three explicit choices:

- every client-facing object path explicitly chooses direct S3 or CloudFront
- the reason for CloudFront is stated, for example cacheability, edge control,
  or origin hiding, rather than "it seemed like the AWS thing to do"
- signed URL TTLs and cache behavior are explicit for client-facing object paths

### 6.3 Large upload posture

For large uploads:

- prefer direct-to-S3 or controlled presigned upload flows
- use multipart upload for large objects
- expire abandoned multipart uploads automatically
- validate size, content type, and ownership intent before granting upload
  capability

Example shared-infrastructure pattern:

- a shared TypeScript CDK AWS infrastructure layer can create a versioned
  service bucket with public access blocked, SSL enforced, and incomplete
  multipart uploads aborted after `7` days
- the same infrastructure layer can route `Object Created` events from that
  bucket into a dedicated SQS queue so downstream object processing runs
  asynchronously in the worker service

## 7. Security Defaults

- S3 automatically applies SSE-S3 encryption to new objects by default; this
  requires no extra bucket configuration
- use SSE-KMS when key-level auditability, cross-account access, or compliance
  posture requires customer-managed or centrally managed keys; SSE-KMS incurs
  per-API-call KMS charges that SSE-S3 does not
- require TLS in transit
- enforce TLS-only bucket access in policy, not only by convention
- block public access at the bucket level unless there is a deliberate and
  reviewed exception
- S3 server access logging is delivered on a best-effort, eventually-consistent
  basis and does not guarantee that every request produces a log entry; for
  compliance or audit requirements that need reliable coverage, AWS CloudTrail
  S3 data events provide per-request logging with stronger delivery guarantees
  at additional cost
- treat cross-account replication and access as security-sensitive changes
- use Object Lock only when a real compliance or WORM retention requirement
  exists; it is not a default product-storage feature

## 8. Reliability And Recovery

- assume duplicate upload attempts can happen
- use deterministic keys or metadata where idempotency matters
- use versioning by default for canonical product buckets when accidental
  overwrite or deletion recovery matters
- enable replication only when there is a defined recovery or geographic
  requirement

Versioning is not free. Turn it on deliberately where recovery value exceeds
the added storage and lifecycle complexity.

Cross-region replication is also not a default. A good baseline stays
single-region and relies on bucket versioning plus database backups unless
stronger RPO or RTO requirements justify more.

## 9. Observability And Cost Controls

Every meaningful S3-backed workflow should monitor:

- request errors
- latency on object operations where it impacts user experience
- bytes stored
- bytes transferred
- anomalous egress
- lifecycle transition and deletion effectiveness
- multipart upload leakage

Cost controls should include:

- aggressive expiry for temporary prefixes
- review of CloudFront vs direct S3 egress posture
- storage-class selection by actual access pattern, not guesswork
- alarms on abnormal storage or transfer growth
- periodic review of buckets and prefixes that are not shrinking as lifecycle
  policy predicts

Before launch, each object family needs an owner, a lifecycle class, a
retention rule, and a cleanup path. Temporary uploads, exports, and other
derived artifacts need explicit expiry, and any client-facing object flow needs
the CloudFront-versus-direct-S3 decision written down.
